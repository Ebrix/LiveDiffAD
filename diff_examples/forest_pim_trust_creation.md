# PIM trust creation between two forests

## Scenario

1. Create a PIM trust between two forests: ADM.LOCAL and DOM.LOCAL.

Use `domain.msc` to do so :
```
This domain: ADM.LOCAL
Specified domain: DOM.LOCAL

Direction:
Incoming: Users in the local domain can authenticate in the specified domain.

Trust type: Forest trust

Transitive: Yes

Outgoing trust authentication level: Forest-wide authentication.

Sides of trust: Create the trust for both this domain and the specified domain.
```

Then add the folowing trust attributes (actions performed on the DOM.LOCAL domain controller):
* Enable SID History
```cmd
netdom trust DOM.LOCAL /domain:ADM.LOCAL /EnableSIDHistory:Yes
```
* Enable PIM trust
```cmd
netdom trust DOM.LOCAL /domain:ADM.LOCAL /EnablePIMTrust:Yes
```

2. Enable the AD PAM feature
```powershell
> Enable-ADOptionalFeature "Privileged Access Management Feature" -Scope ForestOrConfigurationSet -Target adm.local
```
3. Create a Shadow Principal with its `msDS-ShadowPrincipalSid` attribute set to the SID of the domain admins group of the DOM.LOCAL forest
```powershell
> New-ADObject -Type "msDS-ShadowPrincipal" -Name "DOM - Domain Admins" -Path "CN=Shadow Principal Configuration,CN=Services,CN=Configuration,DC=ADM,DC=LOCAL" -OtherAttributes @{'msDS-ShadowPrincipalSid'='S-1-5-21-1047456448-2337630968-2720937775-512'}
```
4. Create an OU (`Shadow Boys`) and a user account (`Jacky`)
5. Add the user account to the Shadow Principal
```powershell
> Set-ADObject -Identity "CN=DOM - Domain Admins,CN=Shadow Principal Configuration,CN=Services,CN=Configuration,DC=ADM,DC=LOCAL -Add @{'member'="<TTL=3600,CN=Jacky,OU=Shadow Boys,DC=ADM,DC=LOCAL>"}
```

## Remarks

### From the point of view of the ADM.LOCAL forest (trusted)

* Creation of a trust accoun of sam account type `SAM_TRUST_ACCOUNT` (UAC: `UF_INTERDOMAIN_TRUST_ACCOUNT`) named with the NETBIOS name of the remote forest concatenate with a "$" member of the domain users group (by its primaryGroupID) (see [[MS-ADTS] section 6.1.6.8.3](https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-adts/d2435927-0999-4c62-8c6d-13ba31a52e1a)
* Creation of a TDO (object class trusted domain) in the Domain/System partition with a CN representing the FQDN of the trusting domain (dom.local). 
We can see the two attributes which contain the cryptographic secrets the two forest use to communicate witch each other (trustAuthOutgoin/trustAuthIncoming those are secret attribute which are never returned).
Here as it's an incoming one-way trust we have the secret of the trust account which is stored in the trustAuthIncoming (see [[MS-ADTS] section 6.1.6.9.1.2 ](https://learn.microsoft.com/en-us/openspecs/windows_protocols/ms-adts/d2435927-0999-4c62-8c6d-13ba31a52e1a)).
* The attribute `msDSEnabled-Feature` of the "PAM Feature" `ms-DS-Optional-Feature` object is set, showing the activation of the PAM feature.
* Shadow principal container is created with its `msDS-ShadowPrincipalSid` attribute set to the SID of the domain admins group, nothing particular to say

### From the point of view of the DOM.LOCAL forest (trusting)

* There is **no** trust account created
* A TDO is created with an ADM.LOCAL CN
* trustAttributes of the TDO are updated when activating the SID history and PIM trust feature
